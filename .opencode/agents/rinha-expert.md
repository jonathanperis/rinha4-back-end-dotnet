## Fraud Specification

Expected labels are generated by exact KNN:

- vectorize transaction to 14 dimensions
- search 3,000,000 reference vectors
- Euclidean distance
- top `k=5`
- `fraud_score = fraud_count / 5`
- `approved = fraud_score < 0.6`
- fraud counts `0`, `1`, `2` approve; `3`, `4`, `5` deny

### Vector Dimensions

Order matters:

1. `amount = clamp(transaction.amount / 10000)`
2. `installments = clamp(transaction.installments / 12)`
3. `amount_vs_avg = clamp((transaction.amount / customer.avg_amount) / 10)`
4. `hour_of_day = hour(requested_at UTC) / 23`
5. `day_of_week = Monday0(requested_at UTC) / 6`
6. `minutes_since_last_tx = clamp(minutes / 1440)` or `-1` when `last_transaction == null`
7. `km_from_last_tx = clamp(last_transaction.km_from_current / 1000)` or `-1`
8. `km_from_home = clamp(terminal.km_from_home / 1000)`
9. `tx_count_24h = clamp(customer.tx_count_24h / 20)`
10. `is_online = 1/0`
11. `card_present = 1/0`
12. `unknown_merchant = 1` if merchant id absent from `known_merchants`, else `0`
13. `mcc_risk = mcc_risk.json[mcc]`, default `0.5`
14. `merchant_avg_amount = clamp(merchant.avg_amount / 10000)`

`-1` sentinel for dimensions 6 and 7 is intentional. Do not replace with `0`.

## Scoring Model

Final score = `p99_score + detection_score`.

Latency:

- `p99 <= 1ms` -> `3000`
- `p99 > 2000ms` -> `-3000`
- otherwise `1000 * log10(1000 / max(p99_ms, 1))`
- every 10x faster -> about `+1000`

Detection:

- weighted errors `E = FP + 3*FN + 5*HTTP_ERR`
- failure rate = `(FP + FN + HTTP_ERR) / N`
- if failure rate `> 15%` -> `-3000`
- otherwise `1000 * log10(1 / max(E/N, 0.001)) - 300 * log10(1 + E)`

Weights mean:

- HTTP error worst
- false negative worse than false positive
- `0` weighted errors gives `3000` detection points

## Optimization Playbook

Before changing perf-sensitive code:

- identify target path: transport, parser, vectorizer, index scan, repair, fallback, LB
- preserve 200 response under load
- preserve top-5 decision semantics or explicitly quantify FP/FN risk
- benchmark with same compose/scorer/env knobs
- archive exact image SHA and env knobs

High-leverage areas:

- reduce bucket candidate scans without changing uncertain cases
- improve risky fallback filters only if public replay and CI stay `0` errors
- keep fast parser compatible with real payload order/shape
- avoid allocations in hot request path
- keep response bytes precomputed
- avoid logging on hot path
- keep socket readiness hardening: LB waits for both UDS files

Be suspicious of:

- fast shortcuts that only passed one public payload replay
- branch/doc mismatches around `SCORER_MODE`
- CI p99 wins that do not transfer to official Mac Mini
- changes that improve mean but hurt p99
- custom LB regressions causing readiness or connection errors
- memory growth above official cap
